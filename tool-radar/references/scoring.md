# 评分框架

## 6 维度（总分 100）

| 维度 | 分值 | 评估要点 | 数据来源 |
|---|---|---|---|
| 功能匹配度 | 25 | 对照用户核心需求维度，覆盖度；缺关键维度扣分 | README / 官网 |
| 项目活跃/健康度 | 20 | 最近 commit 时间、近1年提交频率、release 规律、issue 响应 | gh repo / gh api |
| 用户口碑 | 15 | 社区评价、star/issue 比、负评集中点 | Reddit / 小红书 / social |
| 上手成本 | 15 | 安装步骤数、依赖量、文档质量、是否需后端/账号 | README / 官网 |
| 架构合理性 | 10 | 技术栈成熟度、代码规模、可维护性、设计清晰度 | repo 分析 |
| 生态/扩展 | 15 | 插件机制、二次开发难度、社区贡献、集成能力 | repo / topics |

## 打分细则

- 功能匹配度：完全覆盖核心维度=25；每缺一个关键维度 -5；多余无关功能不扣分。
- 活跃度：近30天有 commit 且 release 规律=20；近1年有提交=12；超1年无更新≤5。
- 口碑：star/issue > 100 且无集中差评=15；有已知严重 bug 或集中差评 -8。
- 上手成本：一条命令安装 + 文档好=15；需配后端/账号 -5；文档缺失 -5。
- 架构：技术栈主流 + 设计清晰=10；冷门栈或代码臃肿酌情扣分。
- 生态：有插件/API/活跃 PR=15；纯单文件无扩展≤5。

## N/A 处理
某维度数据无法获取时标 N/A，该维度从总分剔除，按剩余维度等比折算到 100：
`实得分 = 原始合计 / (100 - N/A维度分值和) * 100`。

## 排名表格式

| 排名 | 工具 | 总分 | 功能 | 活跃 | 口碑 | 上手 | 架构 | 生态 | 一句话定位 |
|---|---|---|---|---|---|---|---|---|---|

- 排序按总分降序；总分相同按功能匹配度降序。
- 每个分数单元格允许写 `20/25` 形式（得分/满分）便于核对。
- 附录逐候选写每维度扣分理由与证据链接/原话。
